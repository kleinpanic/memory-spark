## Root Cause Analysis

### Problem 1: Mock Eval Numbers Are Fiction
The README badges (NDCG@10=0.889, MRR=0.941, Recall@5=0.903) are generated by `evaluation/run.ts --mock`.

**How mock mode works:**
- `mockConfigStrength()` HARDCODES score boosts: rerank +0.12, fts +0.11, context +0.10, etc.
- `mockQueryResult()` generates FAKE retrieved documents with simulated scores
- `buildDistractors()` creates synthetic noise documents
- The "full pipeline" always wins because the code literally adds more points for each enabled feature
- These numbers have ZERO correlation with actual retrieval quality

**Live eval showed:** Full Pipeline NDCG@10=0.284 vs Vanilla NDCG@10=0.560
- The pipeline is ACTIVELY HURTING recall compared to raw vector search
- This is the opposite of what mock mode claims

### Problem 2: All Timestamps Are Wrong
- `pipeline.ts` correctly uses `stat.mtime` (fixed in commit `c547f2c`)
- BUT: the full rebuild earlier today touched all file mtimes via write operations
- Result: ALL 18,647 chunks have `updated_at` within the last few hours
- Temporal decay formula `0.8 + 0.2 * exp(-0.03 * ageDays)` returns ~1.0 for everything
- The `age` field in recalled memories shows "1h ago" for content from February
- This makes temporal decay completely useless

### Problem 3: Garbage In Index
From live audit of the LanceDB table:
- 18,647 total chunks, 99.6% source="memory", 0.4% source="capture"
- 201 session header chunks (# Session: YYYY-MM-DD...)
- 43 Discord metadata chunks (group_channel, sender_id, etc.)
- 389 chunks from default AGENTS.md bootstrap (other agents' "birth certificate" instructions)
- 5,354 chunks from `memory/knowledge-base` (old reference material)
- Quality filter blocks NEW garbage but existing garbage persists in index

### Problem 4: Recall Pipeline Actively Hurts Quality
Why the full pipeline scores WORSE than vanilla:
1. **RRF merge dilutes good results** — FTS returns keyword matches from garbage chunks, which get merged with good vector results and push relevant docs down
2. **Source weighting is backwards** — `learnings.md` gets 0.1x penalty (but it's excluded from indexing anyway so this is dead code), `sessions` gets 0.5x (but sessions aren't indexed either)
3. **Reranker may be hurting** — cross-encoder reranking on noisy candidates can push relevant results below the cutoff
4. **MMR diversity penalizes clusters** — if the top 3 results are from the same document (e.g., AGENTS.md), MMR pushes them apart and inserts irrelevant diverse results

### Problem 5: Context Injection Is Noisy
Klein showed the actual injected `<relevant-memories>` block:
- Old session metadata from months ago being recalled
- Confidence scores of 0.14-0.35 (scraping the bottom)
- Completely irrelevant content (session keys, Discord channel IDs)
- This pollutes the agent's context and makes it DUMBER, not smarter

### Problem 6: Auto-Capture Is Barely Firing
- Only 93 capture chunks vs 18,554 memory chunks
- `minMessageLength` default is 30 chars in config.ts but capture.ts uses `minLen = cfg.minMessageLength ?? 80` — the 80 fallback is too high
- Capture only fires on user messages + assistant messages matching specific patterns
- Most valuable operational knowledge (decisions, fixes) is in assistant messages that don't match the narrow regex
