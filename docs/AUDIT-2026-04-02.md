### 🔴 CRITICAL — Must Fix Before Production

#### C1: FTS WHERE Filter Silently Discarded (Cross-Agent Data Leakage)
- **File:** `src/storage/lancedb.ts`, line 428
- **Validated:** ✅ Confirmed
- **Bug:** `ftsSearch()` declares `const q = ...` then calls `q.where(filters.join(" AND "))` without assigning the result. LanceDB uses an immutable fluent API — `.where()` returns a new query object. The original `q` is unchanged.
- **Contrast:** `vectorSearch()` (line 339) correctly uses `let q` and `q = q.where(...)`.
- **Impact:** All FTS filters (agentId, pool, source, contentType, pathContains) are silently ignored. FTS results from any agent or pool bleed into any agent's recall. In a multi-agent setup, `school` agent could receive `meta` agent memories via FTS.
- **Fix:**
  ```typescript
  // Line 404: change const → let
  let q = this.table.search(query, "fts", "text").limit(limit);
  // Line 428: assign result
  q = q.where(filters.join(" AND "));
  ```
- **Risk:** Low (one-line fix, clear pattern from vectorSearch)
- **Tests needed:** FTS search with agentId filter — verify results only contain that agent's data

#### C2: getState() Init Has No Recovery Path
- **File:** `index.ts`, lines 52–111
- **Validated:** ✅ Confirmed
- **Bug:** If `createEmbedProvider()`, `backend.open()`, or `validateDimsLock()` throws during initialization (e.g., Spark is down at cold start), `initPromise` is set to the rejected promise but never cleared. Every subsequent `getState()` call returns the cached rejected promise immediately. The plugin is permanently dead until gateway restart.
- **Impact:** Transient Spark unavailability at boot (GPU cold start, network blip) permanently bricks all 18 plugin tools.
- **Fix:**
  ```typescript
  initPromise = (async () => {
    try {
      // ... existing init logic ...
      state = { cfg, backend, embed, queue, cachedEmbed, reranker, watcher: null };
      return state;
    } catch (err) {
      initPromise = null; // Allow retry on next call
      throw err;
    }
  })();
  ```
- **Risk:** Low (adds catch to clear promise, preserves existing behavior on success)

#### C4: MAP@k Denominator Incorrect for High-R Datasets
- **File:** `evaluation/metrics.ts`, line 129
- **Validated:** ✅ Confirmed
- **Bug:** Uses `ap / Math.min(totalRelevant, k)` with comment "per BEIR standard." BEIR uses pytrec_eval which divides by `totalRelevant` (total relevant in qrels), not `min(R, k)`.
- **Impact:** Benign for SciFact (avg 1.1 relevant/query → `min(1,10)=1=R`). **Inflates MAP ~3.8× on NFCorpus** (avg 38 relevant/query → `min(38,10)=10` vs correct denominator 38).
- **Fix:**
  ```typescript
  // Line 129: revert to correct BEIR denominator
  scores[queryId] = ap / totalRelevant;
  // Update comment to explain why
  ```
- **Note:** The 2026-03-27 "bug fix" was itself a regression. The original `ap / totalRelevant` was BEIR-correct.

### 🟠 HIGH — Fix Soon

#### H1: Score Clamping Defeats Reranker Gate
- **File:** `src/auto/recall.ts`, line 1019
- **Validated:** ✅ Confirmed
- **Bug:** `r.score = Math.min(1.0, r.score * weight)` with weights like `capture: 1.5`, `mistakes: 1.6`. Any chunk scoring ≥ 0.625 (common for cosine similarity) gets clamped to 1.0. Multiple high-weight chunks all become 1.0, triggering the reranker gate's "tied set" skip (spread < 0.02).
- **Fix:** Apply weighting AFTER reranker gate evaluation, or use additive boost: `r.score = Math.min(1.0, r.score + (weight - 1.0) * scaleFactor)`
- **Risk:** Medium — changes ranking behavior; needs benchmark comparison

#### H2: Embed Tool Calls Unguarded (Crash Risk)
- **File:** `index.ts` — `memory_store`, `memory_mistakes_store`, `memory_rules_store`, `memory_recall_debug`, `memory_temporal`, `memory_related`
- **Validated:** ✅ Confirmed
- **Bug:** Tool execute functions call `s.queue.embedQuery()` or `s.embed.embedQuery()` without try/catch. If Spark goes down mid-session, these throw unhandled.
- **Note:** The EmbedQueue DOES have retry logic (maxRetries: 3) so transient failures are handled. But after retries exhaust, the exception bubbles up unhandled to the gateway.
- **Fix:** Wrap embed calls in try/catch, return graceful error text: `"Embedding service unavailable — try again later."`
