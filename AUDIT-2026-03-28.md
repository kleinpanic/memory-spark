# memory-spark Full System Audit — 2026-03-28

## Executive Summary

The pipeline WORKS mechanically (221/221 tests pass, Tier 2/3 passing), but the **evaluation methodology is fatally flawed**, producing unreliable metrics that have been misleading us for days. The retrieval engine is actually good (BEIR SciFact NDCG@10 = 0.768) — the custom benchmark is measuring the wrong things.

---

## 1. ROOT CAUSE: Why Benchmarks Have Been Inconsistent

### Problem: The Golden Dataset is Garbage

| Issue | Impact |
|-------|--------|
| **Only 24 queries with relevance judgments** | Statistically meaningless — 1 wrong match = ±4% NDCG swing |
| **108 corpus docs from 10 agents** | Index has **25,907 chunks from 21 agents** — corpus covers <5% of the index |
| **11 agents completely missing from corpus** | finance, mika, coach, node, shared, local, lens, claude, codex, legal, pantry — their results ALL count as "false positives" |
| **Binary relevance (0/1/2)** | No graded relevance — can't distinguish "perfect answer" from "somewhat related" |
| **No gold_answer field** | Can't use LLM-as-judge for quality assessment |
| **Path matching fragile** | Corpus uses `~/.openclaw/workspace-meta/MEMORY.md`, index uses `meta:MEMORY.md` — the matching code handles this but edge cases slip through |

### Why We Got "High" Numbers Before
- Previous benchmarks ran against **smaller indexes** (5k chunks) with fewer agents
- Some evaluations used "mock mode" with simulated scores
- The BEIR SciFact baseline (0.768) used a clean academic dataset, not our messy real-world index

### Why Numbers Dropped After Reindex
- Reindex added 25,907 chunks including 11 new agents
- More chunks = more "distractor" documents that the benchmark penalizes as false positives
- But these are actually CORRECT results from agents the golden dataset doesn't know about

## 2. COMPONENT AUDIT

### 2.1 Reranker: `nvidia/llama-nemotron-rerank-1b-v2` ✅ Model is CORRECT

**Live test confirmed it works perfectly:**
```
Query: "What machine does OpenClaw run on?"
Results:
  [0.980] "OpenClaw runs on user, a Debian 13 laptop." ← CORRECT
  [0.025] "The DGX Spark node has a GH200 GPU."
  [0.006] "The weather is nice today."
  [0.001] "Klein uses Exampleville CA as his location."
```

**Why benchmark shows regression (0.455 → 0.327 with reranker):**
The reranker correctly surfaces the BEST match (e.g., `finance:USER.md` for timezone questions), but the golden dataset doesn't include `finance` as a valid agent. So the reranker's good answers are penalized as wrong.

**Conclusion:** The reranker is NOT broken. The evaluation is.

- Model: `nvidia/llama-embed-nemotron-8b` (4096 dimensions)

- Port: 18091, cached embeddings working

## 4. IMPLEMENTATION PLAN

### Phase 1: Fix the Evaluation (MUST DO FIRST)
**Without good evaluation, we're flying blind. Nothing else matters until this is fixed.**

#### 1A. Generate 200+ queries with LLM-as-judge grading
- Use Nemotron-Super-120B on Spark (port 18080) as the judge
- Input: Actual chunks from the index (sample from all 21 agents, all pools)
- Output: JSON with query, relevant_chunk_ids, graded_relevance (0-3), gold_answer
- Evaluation: Retrieve top-10, send to LLM judge → score each result → compute NDCG/MRR

#### 1B. Add multiple BEIR benchmarks
- SciFact (already done: 0.768)
- Add: NQ (Natural Questions), HotpotQA, FiQA, TREC-COVID
- These give us publishable baselines for comparison

#### 1C. Pipeline telemetry
- Log every stage's contribution (vector → FTS → merge → weight → decay → MMR → rerank)
- Per-query breakdown showing where good results get dropped or bad ones promoted

### Phase 2: HyDE Implementation
- Before embedding the query, send it to Nemotron-Super-120B:
  "Given this question, write a short passage that would answer it: {query}"
- Embed the hypothetical document instead of (or alongside) the raw query
- This bridges the query-document vocabulary gap (queries are short, documents are detailed)
- Expected impact: +15-25% NDCG based on published HyDE results

### Phase 3: Parent-Child Chunking
- Small children (200 tokens) for precise vector matching
- Large parents (2000 tokens) returned as context
- Child → Parent lookup via `parent_id` field in LanceDB schema
- This improves both precision (small chunks match better) and recall (big parents give context)

### Phase 4: Reranker Integration Validation
- With proper evaluation, re-benchmark the reranker
- Hypothesis: reranker will show significant improvement once evaluation doesn't penalize correct results from missing agents
- If still underperforming, investigate: top-K window size, document length normalization

### Phase 5: Paper-Ready Benchmarks
- Run all configs against all BEIR datasets + our custom domain eval
- Generate ablation matrix with proper error bars
- Compare against published SOTA (RAPTOR, ColBERT, vanilla RAG, no-memory baseline)
